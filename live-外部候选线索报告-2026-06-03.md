# Live 外部候选线索报告

日期：2026-06-03

## 1. 这轮筛选目标

从 GitHub 公共 open issues 里，挑出更像“真实可服务需求”的候选项，而不是只看关键词匹配。

筛选标准：

1. 需求是否像真实交付，而不是单纯 bug
2. 是否接近你当前主打的三类服务
3. 是否能在不直接打扰对方的前提下，先做本地演练

## 2. 候选项 1

- 标题：`Must have automation to deploy new changes to the cloud`
- 仓库：`G-Gonski/agile-final-project`
- 链接：`https://github.com/G-Gonski/agile-final-project/issues/10`
- 类型判断：自动化脚本 / 部署自动化

为什么值得看：

1. 明确提到 automation
2. 有可交付导向，不是纯粹抱怨式 issue
3. 和你的“自动化脚本代做”主线高度相关

风险：

1. 更偏工程内部任务，不一定是对外可购买服务
2. 公开 issue 的上下文不足，交付边界不完整

本地演练产物：

- brief 文件：`tmp_live_candidate_1.md`
- proposal 文件：`report_assets/live_candidate_1_proposal.md`

## 3. 候选项 2

- 标题：`Need help with workflow setup`
- 仓库：`faisalabdullah-commits/csa`
- 链接：`https://github.com/faisalabdullah-commits/csa/issues/2`
- 类型判断：AI / automation workflow setup

为什么值得看：

1. 直接出现 `Need help`
2. 更接近“服务请求”而不是普通开发 issue
3. 和你当前的 AI workflow setup 服务高度相关

风险：

1. 原始描述很短，信息缺口大
2. 真实预算和紧急程度未知

本地演练产物：

- brief 文件：`tmp_live_candidate_2.md`
- proposal 文件：`report_assets/live_candidate_2_proposal.md`

## 4. 当前结论

这两条都适合当“外部真实线索演练样本”，但还不适合贸然评论或推销。

更现实的用途是：

1. 用它们训练你自己的 proposal 反应速度
2. 借此打磨你对陌生线索的边界判断
3. 积累更像“真实市场需求”的案例样本

## 5. 结论排序

1. `Need help with workflow setup` 更像真实服务线索
2. `Must have automation to deploy new changes to the cloud` 更像内部工程任务，但仍有参考价值
