# 主动外联候选清单

日期：2026-06-03

## 1. 这份清单的用途

这不是“立即发出去”的名单，而是从 GitHub 公共 open issues 里，先筛出**更像真实服务请求**、并且和你当前服务能力更匹配的对象。

核心判断标准：

1. 像不像“可交付需求”，而不是纯 bug
2. 是否贴近你现在能卖的三类服务
3. 是否值得用一段英文私聊/评论去试探

## 2. 第一优先候选

### 候选 A

- 仓库：`faisalabdullah-commits/csa`
- issue：`#2`
- 标题：`Need help with workflow setup`
- 链接：`https://github.com/faisalabdullah-commits/csa/issues/2`

为什么值得试探：

1. 标题里直接出现 `Need help`
2. 更像服务请求，而不是普通开发 backlog
3. 和你当前 `AI workflow setup` 服务高度匹配

主要风险：

1. 原始描述过短
2. 预算、环境、期望输出都不清楚
3. 不一定欢迎陌生外联

当前建议动作：

1. 可以当作第一条“低频但高相关”的试探对象
2. 外联必须短，不要长篇推销

## 3. 第二优先候选

### 候选 B

- 仓库：`G-Gonski/agile-final-project`
- issue：`#10`
- 标题：`Must have automation to deploy new changes to the cloud`
- 链接：`https://github.com/G-Gonski/agile-final-project/issues/10`

为什么值得看：

1. 明确提到 `automation`
2. 指向一个可交付的自动化问题
3. 和你的自动化脚本方向贴近

主要风险：

1. 更像内部工程任务
2. 不一定适合陌生人主动推销
3. 公开上下文不足，边界很模糊

当前建议动作：

1. 更适合作为 proposal 演练样本
2. 不建议优先作为第一次真实外联目标

### 候选 C

### 候选 D

- 仓库：`dativo-io/talon`

- 标题：`Full multi-agent governance (without orchestration)`

### 候选 E

- 仓库：`dativo-io/talon`
- issue：`#154`
- 标题：`Developer onboarding polish (first-run + newcomer docs path)`

为什么只列为观察型：

1. 这些 issue 更像成熟项目的公开 roadmap/backlog
2. 技术内容很强，但不是明显的“外包/服务请求”
3. 主动接触的命中率可能低

它们的价值更在于：

1. 理解市场里哪些 AI/automation 需求正在出现
2. 训练你对复杂项目的 proposal 口径

## 5. 当前外联建议

如果你真的要试一次陌生外联，我建议顺序是：

1. `csa#2`
2. 不急着碰 `agile-final-project#10`
3. `talon` 这类先观察，不主动碰

## 6. 不要做的事

1. 不要给一堆公共 issue 群发模板化推销
2. 不要对明显是内部 roadmap 的 issue 硬推服务
3. 不要一上来就发长 proposal
4. 不要把 proposal 草稿直接贴到别人的 issue 里

## 7. 你现在最现实的动作

1. 继续用这类候选 issue 训练 proposal 反应速度
2. 如果真要发，先只试 1 条最像服务请求的对象
3. 先用极短试探话术，而不是直接卖
